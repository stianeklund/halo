# Handover: Model Allocation Strategy for the Halo Decomp Agent Fleet

**Audience:** a Claude Fable 5 instance asked to design/refine which model runs
which kind of work in this project — and, by extension, in the user's agent
workflows generally.
**Written:** 2026-07-04, by the Fable 5 instance that ran the scenario.obj
goal-lift session in `/mnt/g/dev/halo-clean-main`.

## 1. What this project is

Faithful, evidence-based decompilation of Halo CE Xbox (`cachebeta.xbe`) back
to C89 that byte-matches the original MSVC 7.1 codegen. The unit of work is a
**lift**: pick a function, reverse it in Ghidra, write C that reproduces the
original ABI/layout/side-effects exactly, then verify with:
- **VC71 verify** — recompile with the original compiler, instruction-diff
  against a delinked reference object (score = % insns matched; ≥90% is the
  commit bar, ≥99% is "done").
- **Unicorn equivalence** — run oracle vs candidate object in an emulator over
  seeded inputs; compare returns, memory writes, and stubbed-callee args.
- **A fail-closed review gate** — an opus reviewer agent that must return
  AUTO_ACCEPT before anything commits.

Work is orchestrated by a Workflow script
(`/mnt/g/dev/halo-clean-main/.claude/workflows/goal-lift.js`) that spawns one
subagent per pipeline stage: Select → Research → Lift → (redelink, permute,
escalate) → Review → equiv → Commit/Revert/Park → Report. Each stage's
`agent()` call takes a `model:` override — that's the lever this document is
about. Repo doctrine lives in `CLAUDE.md` + `.claude/skills/lift-*`; session
memory in `~/.claude/projects/-mnt-g-dev-halo/memory/`.

## 3. Empirical evidence from this project (don't re-derive — reuse)

These are measured outcomes from the 2026-07-03/04 goal-lift runs, not theory:

1. **Sonnet on long-form code generation is a hard failure here.** The lift
   agent on sonnet was stall-killed by the harness's 180s silence watchdog
   **six consecutive times** (11–50 min each, ~2h total, zero output), and
   because a killed agent returns nothing, the workflow retried the identical
   cache key forever. Opus completed the same lift first try. (Memory:
   `feedback_workflow_sonnet_lift_stall_loop.md`.) Rule: never put sonnet on a
   stage that blocks the pipeline and generates long code/edits.
2. **Opus is the right tier for lifting and reviewing.** Steady-state ~20–40
   min per function through the full pipeline; lifts landed at 93% and 99.5%.
3. **A second opinion from the same tier is waste.** The opus review gate
   returned NEEDS_RUNTIME on an 85.8% lift, equivalence then passed 100/100
   seeds, and a *second* opus review re-rejected on the same structural
   grounds — three expensive runs to produce a verdict a one-line rule could
   have produced. Fixed 2026-07-04: the re-review was replaced by a mechanical
   acceptance rule in `goal-lift.js`.
4. **Most stages aren't judgment at all.** Redelink, equivalence runs,
   permuter runs, commit (message is generated by a script), revert, park,
   progress-log: the prompt already contains the exact shell commands; the
   agent executes and reports structured output. Moved to haiku 2026-07-04.
5. **Fable earns its cost on system design and forensic root-cause, not
   volume.** In this session fable-tier work was: diagnosing the stall-loop,
   redesigning the workflow (model tiering, parking policy, mechanical accept
   rule, equiv intrinsic-filter fix), and calling a 100-seed equivalence
   "failure" a harness artifact (oracle-only `_ftol2` intrinsic calls that
   clang inlines) — a judgment that rescued a correct lift from the discard
   pile. The workflow reserves fable for one slot: **lift escalation** when an
   opus lift lands at 65–84% without a known structural cap.

## 4. Current allocation in goal-lift.js (as of 2026-07-04)

| Stage | Model | Why |
|---|---|---|
| select | sonnet | Filter/triage of a tool's output; advisory, non-blocking-ish |
| research (RE brief) | opus | Feeds ABI decisions; wrong = corrupted lift |
| lift attempt 1 | opus | Long code generation; sonnet stalls (see §3.1) |
| lift escalation (65–84 uncapped) | **fable** | Only slot where 10x intelligence changes the outcome |
| review gate | opus | Fail-closed correctness gate; never downgrade |
| redelink / equiv / permute / commit | haiku | Fixed commands + structured report |
| revert / park / progress-log | haiku | Mechanical |
| NEEDS_RUNTIME adjudication | **no model** — mechanical rule | score ≥85 + equiv pass at moderate+ confidence → accept; weak → park |

## 5. Transferable principles (the actual deliverable)

1. **Match the model to the cost of being wrong, not to task prestige.**
   Stages whose errors are cheap and self-evident (a failed command, a build
   error) go to the cheapest model. Stages whose errors are silent and
   expensive (ABI misjudgment, accepting a wrong lift, discarding a right one)
   get the strongest model — or better, a mechanism.
2. **Mechanism beats model.** Before asking "which model should judge this?",
   ask "can code judge this?" A deterministic rule is cheaper, faster, and
   more consistent than any model, and it never re-litigates. Models should
   only consume ambiguity, not re-confirm evidence.
3. **Don't buy the same judgment twice.** Two same-tier opinions on the same
   evidence correlate; the second adds cost, not signal. If a verdict needs
   strengthening, add *new evidence* (a runtime test, a disassembly diff), not
   a new opinion.
4. **Escalate on failure, never start high.** Ladder: haiku (execute) → sonnet
   (triage/explore) → opus (generate/review) → fable (design, forensics,
   plateau-breaking). A stage failing twice at tier X is evidence for X+1;
   retrying at X is the definition of waste. De-escalate when a stage's
   prompts have matured to the point of being scripts — that's how
   redelink/commit fell from sonnet to haiku here.
5. **Sonnet's niche (this user's genuine uncertainty):** read-heavy,
   latency-tolerant, advisory work — candidate triage, codebase exploration,
   summarizing tool output, first-pass doc drafts. Anything where a stall or a
   mediocre answer costs a retry, not a pipeline. If a sonnet stage ever
   blocks a serial pipeline, promote it or make it async.
6. **Fable's niche:** (a) designing/repairing the *system* — workflows,
   verification methodology, acceptance policy; (b) forensic root-cause where
   a wrong conclusion costs days (silent miscompiles, memory corruption,
   "is this failure real or a harness artifact?"); (c) breaking plateaus the
   opus tier has demonstrably failed at. One fable hour that fixes a policy
   pays for itself across every future run — fable on volume work never does.
7. **Watchdogs change the calculus.** This harness kills agents silent >180s.
   Model choice must account for *harness dynamics*, not just capability:
   a cheaper model that streams steadily can beat a smarter one that thinks
   silently past the watchdog. Wrap long shell commands in `timeout`.

## 6. Open items the next instance should know about

- **Parallel lifts (designed, not implemented):** lift stages serialize on the
  shared working tree (~35 min/function wall-clock). Design: per-candidate
  `isolation: 'worktree'` lift agents (3-way apply of result patches back in
  the main tree before commit). Validated in miniature 2026-07-04 by running a
  score-improvement agent in a worktree concurrently with the live run.
- **Parked near-miss queue:** 85–89% lifts blocked at the review gate are now
  parked, not reverted — patches + `ledger.jsonl` in
  `/mnt/g/dev/halo-clean-main/artifacts/auto_lift/parked/` (4 entries:
  FUN_0018b190 87%, FUN_0018dcf0 86.4%, FUN_0018ef30 85.8%, FUN_0018b990 ?).
  Treat as a work queue; memory `feedback_park_near_miss_lifts.md`.
- **Equivalence harness fix (2026-07-04):** `tools/equivalence/stubs.py` now
  filters MSVC compiler-runtime intrinsics (`_ftol2`, `_allmul`, SEH pro/epi
  logs, …) from the stub call-seq differential — the oracle calls them, clang
  inlines them, and the old comparison failed correct lifts on that alone.
  `FUN_001daf7e` (also oracle-only in dcf0's trace, LIBCMT address band) is
  unconfirmed — resolve before trusting dcf0's equiv verdict.
- **Live workflow runs use a frozen script copy.** Edits to the repo's
  `goal-lift.js` apply to *future* runs only; never sync edits into a live
  run's session script (it invalidates the resume cache).
- **Two checkouts:** `/mnt/g/dev/halo` (user's main) and
  `/mnt/g/dev/halo-clean-main` (where this session's runs, parked artifacts,
  and workflow edits live). Never run git mutations across checkouts.
